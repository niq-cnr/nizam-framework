Problem → Solution

Four named design decisions remediate four recurring flaws in agent-driven engineering pipelines. Each is binding on the framework's own implementation.

DD-1 — Tool-Driven State Management

Flaw remediated: context exhaustion from bulk-reading governance directories.

Agents route through the root NIZAM.json capability index (validated against registry/nizam-index.schema.json) to locate only the minimal governance files a task needs, instead of reading an entire standard/, methodology/, templates/, or tools/ directory "just in case." Authoritative in methodology/04_tool_driven_state.md.

DD-2 — Universal Circuit Breaker

Flaw remediated: infinite retry loops.

Every execution loop embeds a mandatory 3-strike circuit breaker: three failed attempts on a single step halts it, marks it BLOCKED, logs the failure, and escalates to a human — a fourth attempt is forbidden. Authoritative in methodology/03_circuit_breaker.md.

DD-3 — Evidence Externalisation

Flaw remediated: brittle proof pasted inline into YAML or JSON strings.

Every piece of proof that a step succeeded is written to its own file under .agent/evidence/ and referenced by path, never pasted inline. See methodology/04_tool_driven_state.md Section 3.

DD-4 — Unified Skill Payload

Flaw remediated: duplicated, per-runtime skill forks.

Exactly one skill payload ships under tools/ — no .claude/, .codex/, or other per-runtime fork directories. Any runtime loads the same tools/SKILL.md through the adapter contract in tools/interface.md.

Quickstart

Governance inheritance is one atomic clone → inject → verify operation, implemented once as bootstrap.sh at the framework's repository root — an evolution of AGIP (the earlier prototype this framework evolved from, per bootstrap.sh's own header comment) into a single, reusable, runtime-agnostic script. From a checkout (or vendored copy) of the framework at a pinned tag, run:

GOVERNANCE_TAG=v0.1.0 ./bootstrap.sh --tag v0.1.0

GOVERNANCE_TAG (or the equivalent --tag flag) is required and never defaulted: an empty tag, main, master, HEAD, or any refs/heads/* branch reference is refused, because a floating branch is never an acceptable inheritance pin (standard/GIP.md Section 2, point 1).

This clones the pinned tag, stages the governance payload, verifies it landed correctly, and only then atomically installs it under the target directory (.nizam/ by default). What you get:

.nizam/
├── standard/
├── templates/
├── schema/
├── tools/
├── NIZAM.json
└── provenance.json

To check an already-bootstrapped target for drift, with no network access and no write, run:

GOVERNANCE_TAG=v0.1.0 ./bootstrap.sh --verify-only --tag v0.1.0

--verify-only confirms the target directory exists, its NIZAM.json parses, every path it indexes resolves on disk, and its recorded provenance.json tag matches the expected pinned tag — performing no clone and no write.

Adoption Paths

New repository

Per the Governance Inheritance Protocol (GIP) (standard/GIP.md Section 2): clone a pinned semantic-version git tag of the framework (never main or any floating branch), inject standard/, templates/, schema/, and tools/ together with the root NIZAM.json capability index into your workspace, load the injected documents into working context before acting, and record the pinned tag as your inheritance baseline. bootstrap.sh performs this as one atomic clone → inject → verify operation (Section 2.1).

Existing repository

A repository is rarely empty when it first adopts Nizam. standard/GIP.md Section 5 governs coexistence:

  • Conflict rules. A pre-existing CONTEXT.md, AGENTS.md, or CI (continuous integration) configuration is never silently overwritten. Bootstrap injects only standard/, templates/, schema/, tools/, and NIZAM.json; a colliding pre-existing file is preserved under a renamed path (for example CONTEXT.md.pre-nizam) so a human or agent can diff it against the framework's templates/ version and reconcile by hand. Existing CI is reconciled — the recommended compliance checks are added to it, not substituted for it.
  • Tier 1 — docs-standard-only. Inherit standard/ alone. Agents read and are bound by the governance documents; no Generator/Validator/Evaluator cycle runs yet.
  • Tier 2 — templates. Additionally inherit templates/, so the repository can author its own governed CONTEXT.md, AGENTS.md, ADRs, and other project documents, still without running the contract loop.
  • Tier 3 — full loop. Inherit the complete governance payload and run the complete contract-first Generator/Validator/Evaluator cycle described in methodology/01_execution.md, end to end.

The Execution Loop

Under the Agent Governance Framework (AGF) (standard/AGF.md), every feature moves through a two-loop, contract-first state machine (methodology/01_execution.md) before it is considered complete:

  1. Planner expands a request into a specification and feature list.
  2. Generator proposes a contract (status: "proposed") scoping the feature's deliverables, non-goals, and verification commands.
  3. Validator Mode A — the pre-code contract gate — confirms the contract is complete, bounded, testable, and traceable to the specification.
  4. Evaluator contract review — an independent second read of the same contract.
  5. Generator implements only the approved contract's scope.
  6. Validator Mode B — the post-code implementation gate — diffs the resulting change set against the approved contract, confirming only contracted files were touched and nothing is a stub.
  7. Evaluator QA verdict — independently executes the contract's verification commands and records a pass/fail verdict with evidence.

Validator Mode A, Validator Mode B, the Evaluator's contract review, and the Evaluator's QA verdict each culminate in a machine-parseable JSON verdict; advancement is permitted only when approved is true and issues, missing_acceptance_coverage, and unsupported_claims are all empty (standard/AGF.md Section 4).

Every repeatable step above — a contract revision after a Mode A or contract-review rejection, or an implementation rework after a Mode B or QA rejection — is bound by the framework's mandatory 3-strike circuit breaker (methodology/03_circuit_breaker.md): attempt 1 is a direct fix, attempt 2 a type/interface analysis, attempt 3 an architectural review, and attempt 4 is forbidden — the step's working changes are discarded, the step is marked BLOCKED, the failure is logged to the technical-debt register, and the pipeline halts for human review.

Anatomy of a Governed Repository

A repository governed by Nizam communicates results exclusively through durable, file-based state — never through chat history or conversational memory (the durable-state rule, standard/AGF.md Section 5). Every run maintains its position through file families under .agent/:

.agent/
├── run_state.json      # current phase/feature position, status, circuit-breaker counters, history
├── feature_list.json   # the planned feature DAG: dependencies and acceptance tests
├── contracts/
│   └── NNN.json        # per-feature scope: deliverables, non-goals, verification commands
├── qa/
│   └── NNN.json        # the Evaluator's independent pass/fail verdict, with evidence
└── evidence/
    └── *.txt           # captured terminal output, one file per verification command

Lifecycle: a Planner writes feature_list.json; a Generator writes a proposed contracts/NNN.json; once both Validator gates approve, the Generator implements and captures proof to evidence/; the Evaluator writes the corresponding qa/NNN.json verdict; run_state.json then advances to the next eligible feature. See methodology/04_tool_driven_state.md Section 4.

Agent Integration

An agent runtime discovers the skill payload per tools/interface.md Section 2, in this order:

  1. Bootstrapped-consumer discovery. First look for tools/skill.json under the injected governance target directory — by default .nizam/tools/skill.json.
  2. Framework-checkout fallback. If no .nizam/tools/skill.json is found, fall back to a repository-root tools/skill.json — the layout this framework repository ships at its own root for a framework checkout evaluating itself.
  3. Pinned-checkout discovery. Where a consumer instead references a pinned release tag without vendoring files locally, resolve tools/skill.json inside a checkout of that pinned tag.

Once tools/skill.json is located, every protocol this skill summarizes assumes an agent runtime can perform exactly three abstract operations (tools/interface.md Section 4), each mapped by the runtime's own adapter onto its native tool-calling primitives:

read-state
Read the current durable-state content a task needs — NIZAM.json, a governance module file, or a .agent/ artifact.
write-evidence
Persist a material result — a document, a durable-state update, or captured verification output — to its canonical durable location.
run-verification
Execute a contract's verification commands and capture the command's output and exit code as evidence.

An integrator adapting a new runtime checks conformance against tools/interface.md Section 5's numbered checklist, condensed here: the adapter locates tools/skill.json, loads its entry_point (tools/SKILL.md) content verbatim rather than a paraphrase, injects it as instructions-tier context rather than ordinary conversation, loads it conditionally on the triggers tools/SKILL.md Section 1 defines, maintains no forked or rewritten copy of it, backs all three abstract operations above with a real file-read, file-write, and command-execution primitive respectively, surfaces every operation's errors rather than suppressing them, and does not fabricate governance behavior when no tools/skill.json can be discovered.

FAQ

What is Nizam not?
Nizam is a pure governance / methodology payload. It ships standards, protocols, schemas, and templates — it does not ship application code, infrastructure, or runtime services.
How is a version pinned?
Consumers clone a specific semantic-version git tag (for example v0.1.0), never main or any other floating branch reference (standard/GIP.md Section 2, point 1).
How is drift detected, and how is it fixed?
A consumer periodically re-verifies its injected governance content against its recorded pinned tag, checking for local hand-mutation and staleness against newer releases (standard/GIP.md Section 4). The correct remediation is always to re-run bootstrap.sh against the appropriate tag — hand-patching an individual injected file is prohibited, because it reintroduces exactly the divergence drift detection exists to catch.
What license does Nizam ship under?
The license choice is a separate, human-gated feature and has not landed as of this guide's publication. This guide does not assert a specific license.
