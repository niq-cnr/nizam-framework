# Technical Debt Register

## Open

| ID | Date | Severity | Description | Remediation |
|----|------|----------|-------------|-------------|
| NDEBT-007 | 2026-07-12 | Medium | `tools/skill.json` content is validated by nothing in CI: C4 JSON-parses only `NIZAM.json`, C9 sweeps only `.md`/`.html` bodies, C5's sweep set excludes it, and the e2e harness asserts only the file's existence. Consequence: its `release_train` capability shipped pointing at the retired `methodology/05_release_train.md` from v0.4.0 through v0.5.3 undetected, while the identical stale pointer in `tools/interface.md` (an `.md` file, swept) WAS caught and fixed in the same renumbering cleanup. The pointer itself is fixed as of this entry; the enforcement gap is not. | Extend the validator (a new check or a C4 extension) to JSON-parse `tools/skill.json`, resolve every `capabilities[].module` path and the `entry_point`, in default and `--payload` modes; add a matching negative fixture. |
| NDEBT-008 | 2026-07-12 | Medium | The bootstrap-injected payload references files it does not inject: 6 of 10 `tools/skill.json` capabilities and multiple `tools/SKILL.md` cross-references point at `methodology/*`, but `bootstrap.sh` injects only `standard/`, `templates/`, `schema/`, `tools/`, and `NIZAM.json`. In a consumer `.nizam/` install those paths dangle; `bootstrap.sh`'s own index verification deliberately skips `methodology/` paths, so inject→verify passes without noticing. Related consumer-context concern: NDEBT-004. | Decide explicitly: either add `methodology/` to the injected payload (GIP/README/CONTEXT payload enumerations and C4/C9 payload-mode rules all update together), or declare pinned-framework-checkout resolution the canonical answer for `methodology/*` references in consumer installs and state that in `tools/interface.md`'s discovery contract — then make the e2e assert whichever contract is chosen. |
| NDEBT-009 | 2026-07-12 | Low | The negative fixtures under `tools/fixtures/` are substantive and discriminating (each fails its targeted check when run via `--target`), but dormant: no CI job runs them, so a regression that made a validator check vacuously pass would not be caught. The full-sweep CI runs exclude `tools/fixtures/` by design. | Add a fixtures CI job (or a `validate.sh --self-test` mode) that runs every fixture through its targeted check and asserts the expected non-zero exit, alongside the existing `validate` and `e2e_bootstrap` jobs. |
| NDEBT-010 | 2026-07-12 | Medium | The Orchestrator role is load-bearing but undefined: `standard/AGF.md` Section 2 defines exactly four roles (Planner/Generator/Validator/Evaluator), yet `standard/capability_profiles.md` ships an `orchestrator-primary` profile while claiming every profile maps to an AGF role, `standard/permission_classes.md` and `standard/mcp_policy.md` allocate orchestrator permissions and tool surfaces, and `methodology/06_release_train.md` concedes AGF defines no release-manager role. The framework's own run history is orchestrator-driven throughout. | Define the Orchestrator (coordination/state-ownership role, its gate obligations, and which `run_state.json` fields it exclusively owns per AGF Section 5 rule 4) in AGF Section 2 — a deliberate standards change to route through the next planned phase, not a hot-patch. |
| NDEBT-011 | 2026-07-12 | Low | `templates/work-packet.template.json` does not validate against `schema/work-packet.schema.json` (three errors: `{{TIER}}` and `{{BLAST_RADIUS}}` placeholder tokens occupy enum-constrained fields; `{{MERGE_ORDER}}` occupies an integer field), while the schema's own `description` claims it validates the template. `templates/README.md` quietly downgrades the template's contract to JSON parse-validity only; the two claims disagree. | Align the surfaces: either give the template schema-valid literal defaults (e.g. `tier-0`, `local`, `0`) so it validates end-to-end, or correct the schema `description` to name work-packet *artifacts* rather than the template and keep the parse-validity-only contract, stating the choice in `templates/README.md` (the `schema/README.md` table row added 2026-07-12 already carries the caveat). |
| NDEBT-004 | 2026-07-10 | Low | C9 in `--payload` mode is proven non-breaking on the framework's OWN tree only. A real bootstrapped consumer whose `.nizam/` payload lacks injected `methodology/` and `docs/architecture/` could false-fail on directory-qualified cross-references, because C9 — unlike C4 — has no payload `skipped_dirs` carve-out. Surfaced by the F-026 Mode-A validator; the F-028 QA independently observed C9 legitimately fails in a `.agent/`-stripped consumer context. | DEFERRED beyond F-029. F-029 (the hermetic e2e harness) is spec-scoped (product_spec_004 §4.2) to bootstrap.sh's OWN `--verify-only` inject→verify assertions (payload present, index-valid, `.nizam/tools/skill.json` discovery, verify-only exit 0), NOT `validate.sh --payload` — that would require editing C9 (a different check, out of F-029's scope; Mode A ruled the NO-decision spec-faithful). The `validate.sh --payload` consumer-safety diagnostic + any C9 payload `skipped_dirs` carve-out is deferred to a future self-compliance feature. |
| NDEBT-005 | 2026-07-10 | Low | C10's three spec-scoped sub-checks (payload-set / discovery-order / version-anchor) do NOT mechanize a recurrence-guard for the two NDEBT-003 defect CLASSES that F-027 fixed by hand: (a) enumeration-completeness (a future doc could again omit a `methodology/`/`registry/` file from a hand-maintained key-documents list) and (b) bare (non-`/`-qualified) stale cross-references (neither C9's `/`-qualification nor C10's three sub-checks catch a bare `NN_name.md` that no longer resolves).
