--- FIXTURE A: external-URL check (verification index 4) single-quote blind spot ---
Fixture text: <a href='https://evil-tracker.example.com/beacon'>bad single-quoted link</a>
Command under test (verbatim regex from contract):
  urls = re.findall(r'(?:href|src)="(https?://[^"]+)"', text)
  bad = [u for u in urls if not re.match(r'https?://(github\.com|niq-cnr\.github\.io)/', u)]
Result: urls=[] bad=[] -> script prints "PASS (urls found: [])"
EXIT:0
FINDING: the regex only matches double-quoted href/src attribute values. A disallowed
external URL written with single quotes (or unquoted) is invisible to this check and the
verification FALSE-PASSES. Confirmed reproducible.

--- FIXTURE B: path-resolution check (verification index 8) trailing-period false-fail ---
Fixture text: "For details, read docs/architecture/ADR-002-html-user-guide.md."
Command under test (verbatim regex from contract):
  paths = set(re.findall(r'(?:standard|methodology|registry|templates|schema|tools|docs)/[A-Za-z0-9_./-]+', text))
Result: paths = {'docs/architecture/ADR-002-html-user-guide.md.'}
FINDING: the character class [A-Za-z0-9_./-] includes '.', so a real, existing path named
in ordinary prose immediately followed by a sentence-ending period is captured WITH the
trailing period appended, producing a path string that does not exist on disk
("...md." vs the real "...md"). This is a false FAIL for the right file referenced the
wrong way — the check would reject a guide that correctly names a real path in body prose
without wrapping it in <code>/href. Confirmed reproducible.

--- FIXTURE F: scope-guard (verification index 15) blind to brand-new untracked files ---
Setup: a git repo with one committed tracked file (tracked.txt). Then: (a) tracked.txt is
modified, (b) a brand-new directory docs/guide/index.html is created (untracked, exactly
mirroring this feature's primary deliverable), (c) a brand-new NIZAM.json is created
(untracked, standing in for "NIZAM.json modified" -- in the real repo NIZAM.json is already
tracked so it WOULD show; this sub-case isolates the untracked-file behavior only).
Command under test: git diff --name-only HEAD -- . ':(exclude).agent'
Result: only "tracked.txt" (and, in one run, another pre-existing tracked file) appeared.
Neither the new docs/ directory nor the new file were listed, despite `git status --short`
showing them with `??`.
FINDING: `git diff --name-only HEAD` never reports brand-new (untracked) files, only
modifications to already-tracked paths. In the real feature-018 change set, docs/guide/index.html
is a NEW file (untracked at verification time) and NIZAM.json is an EXISTING tracked file
being modified. Therefore this scope guard can only ever observe/allowlist NIZAM.json; its
"docs/guide/index.html" branch of the allow-list is dead code, and -- more importantly -- the
check provides NO protection against the generator creating any OTHER new, out-of-scope
untracked file anywhere in the tree (e.g. a stray docs/guide/style.css, a scratch file, or an
extra doc). Such a file would silently pass this "scope guard" while genuinely violating
scope. Confirmed reproducible; not a hypothetical.
